# Matchers Specification

## Purpose
Verifies that an expected security alert was generated by the SIEM in
response to a scenario detonation. Matchers poll the SIEM during the
scenario's expectation timeout, returning success on first match. Two
implementations exist: `ElasticSecurityAlertMatcher` and
`DatadogSecuritySignalMatcher` (covered by sibling specs). This spec
captures the umbrella contract — polling behavior, the "indicator
substring" match strategy, optional cleanup, and the explore-mode hook.

## Requirements

### Requirement: Matcher Interface
The system SHALL define matchers as objects implementing
`HasExpectedAlert(ctx) -> (matched bool, evidence, error)` and optionally
`Cleanup(ctx) -> error`. The runner SHALL call `HasExpectedAlert`
repeatedly during the scenario's timeout, treating the assertion as
satisfied on the first `matched = true` return.

#### Scenario: Match on first hit
- **WHEN** the matcher returns `matched=true` on the second poll iteration
- **THEN** the runner records the assertion as passed and stops polling that assertion

### Requirement: Polling Cadence
The system SHALL poll matchers every 2 seconds during a scenario's
expectation timeout. The interval SHALL be the same for all matcher
types and is not configurable per-scenario.

#### Scenario: Poll interval
- **WHEN** a scenario has multiple expectations and is in matching phase
- **THEN** each matcher's `HasExpectedAlert` is called approximately every 2 seconds until it matches or the timeout fires

### Requirement: Round-Robin Across Multiple Assertions
The system SHALL check unmatched assertions on every poll iteration; a
matched assertion is not re-checked. A single slow assertion SHALL NOT
block other assertions from being checked.

#### Scenario: Mixed match progress
- **WHEN** assertions A, B, C exist and only A matches first
- **THEN** B and C continue to be polled every 2 seconds while A is no longer queried

### Requirement: Timeout Surfaces Unmatched Assertions
The system SHALL return an error from the assertion phase when one or
more assertions remain unmatched at timeout, listing each unmatched
assertion's name in the error message.

#### Scenario: Partial match at timeout
- **WHEN** assertions A and B exist and only A matched before the 5-minute timeout
- **THEN** the scenario result records B as failed and the runner's error message includes "B"

### Requirement: Indicator-Substring Match Strategy
The system SHALL determine a match by serializing each candidate alert /
signal to JSON and checking whether ANY indicator string appears as a
substring (`strings.Contains`) anywhere in the serialized representation.
**Note:** this is intentionally broad and may produce false positives
for short or common indicator values; flagged as a known design choice.

#### Scenario: Execution ID found in any field
- **WHEN** an Elastic alert document contains the run's execution UUID anywhere in its `_source`
- **THEN** the matcher considers the document a match

### Requirement: Indicator List Composition
The system SHALL build the indicator list from: the detonation's
`execution_id` (always included), `execution_uuid` if present in
detonation output, all entries in the YAML `indicators.static` array,
and Terraform output values keyed by `indicators.terraformOutput`.

#### Scenario: Static and execution ID
- **WHEN** a scenario has `indicators: {static: ["malicious.example.com"]}` and detonation produces `execution_id: "abc"`
- **THEN** matchers look for `"malicious.example.com"` and `"abc"` in candidate alerts

### Requirement: Cleanup Closes or Archives Matched Alerts
When invoked, the system SHALL transition matched alerts to a "closed" /
"archived" state via the SIEM's API. Cleanup is opt-in per run via
`RunRequest.cleanupAlerts`. Cleanup errors SHALL be logged but SHALL NOT
fail the scenario result.

#### Scenario: Cleanup after match
- **WHEN** a scenario passes with `cleanupAlerts: true`
- **THEN** the matched alerts are transitioned to closed/archived in the SIEM and the scenario result remains successful

### Requirement: Lazy Credential Initialization
The system SHALL allow matcher construction at parse time without
credentials and SHALL only attempt to read credentials from the run
environment on the first `HasExpectedAlert` or `Cleanup` call. **Note:**
this allows lint to construct matchers without contacting any SIEM.

#### Scenario: Lint without creds
- **WHEN** the lint endpoint parses a scenario with an Elastic expectation and no credentials in env
- **THEN** parsing succeeds and the matcher is constructed for shape validation only

### Requirement: Explore Mode Bypasses Match Logic
The system SHALL, when `RunRequest.exploreMode = true`, query all open
alerts/signals (without rule-name filtering) at each poll, accumulate
those whose serialized form contains any indicator string into
`DiscoveredAlerts`, and run the full timeout without recording pass/fail
outcomes for individual assertions. At the end of the run, the scenario
SHALL be marked successful (`isSuccess = true`) when at least one alert
was accumulated into `DiscoveredAlerts`, and marked failed
(`isSuccess = false`) with a descriptive error message when zero alerts
were discovered.

#### Scenario: Explore run with discoveries
- **WHEN** a 1-minute explore-mode run starts and 4 distinct alerts contain the execution ID at various points
- **THEN** all 4 are recorded in the result's `DiscoveredAlerts`, no per-assertion pass/fail is recorded, and the scenario's `isSuccess` is `true`

#### Scenario: Explore run with no discoveries
- **WHEN** an explore-mode scenario completes its full timeout without any alert containing any of the configured indicators
- **THEN** `DiscoveredAlerts` is empty, the scenario's `isSuccess` is `false`, the scenario's error message indicates that no matching alerts were found, and the parent run's `failed` counter includes this scenario

### Requirement: Per-Type Time Window Pinning
The system SHALL pin search time windows differently per matcher
implementation: Elastic matchers SHALL anchor to the run start time when
`SetSince` is called by the runner; Datadog matchers SHALL use a fixed
1-hour lookback. **Note:** this asymmetry is captured in per-type specs.

#### Scenario: Elastic pinned to start
- **WHEN** a scenario starts at T and runs the Elastic matcher
- **THEN** the matcher's query filters `@timestamp >= T`

### Requirement: Match Result Captured On Scenario Result
The system SHALL persist each assertion's outcome in the
`scenario_results.assertions` JSONB field, including the matcher type,
assertion name, pass/fail flag, and matching alert evidence (for the
matched alert) or absence-of-match evidence (for failed assertions).

#### Scenario: Successful assertion captured
- **WHEN** an assertion passes
- **THEN** the scenario_result's `assertions` JSON includes that assertion with success=true and the matched alert's identifier
